# Хранилище данных Legday

В проекте нет отдельной СУБД (SQLite, Core Data). Все данные хранятся в **UserDefaults** (ключ–значение). Ниже описаны логические «таблицы» (группы ключей и структур), поля и связи между ними.

---

## Общая схема

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         UserDefaults.standard                           │
├─────────────────────────────────────────────────────────────────────────┤
│  journalEntries (Data/JSON)     →  массив JournalEntry                  │
│  timer_*                        →  состояние таймера (фаза, конец, пауза)│
│  stands_<date>                  →  счётчик вставаний за день            │
│  standingMin_<date>             →  минуты стоя за день                  │
│  streakDays                     →  текущая серия дней                    │
│  lastStreakDay                  →  дата последнего дня серии             │
│  lastResetDay                   →  день последнего сброса счётчиков      │
│  reminderIntervalMinutes, ...  →  настройки (SettingsStore)            │
└─────────────────────────────────────────────────────────────────────────┘
```

Связи:
- **Дата дня** (`yyyy-MM-dd`) связывает дневную статистику (`stands_<date>`, `standingMin_<date>`) с календарём и с расчётом серии (`lastStreakDay`).
- **Журнал** не имеет внешних ключей; по полю `type` и `date` можно считать вставания/приседания и минуты стоя за день.
- **Таймер** и **настройки** не ссылаются на другие сущности; настройки задают интервалы, которые использует таймер.

---

## 1. Журнал (Journal)

**Ключ в UserDefaults:** `journalEntries`  
**Тип значения:** `Data` (JSON-массив объектов `JournalEntry`)  
**Код:** `StandUpState.swift` — `JournalEntry`, `loadJournal()`, `saveJournal()`, `addJournalEntry`, `removeJournalEntry`, `clearAllHistory`.

Хранится один массив записей; при количестве > 500 старые удаляются с конца.

### Поля (структура JournalEntry)

| Поле | Тип | Обязательное | Описание |
|------|-----|--------------|----------|
| **id** | UUID | да | Уникальный идентификатор записи. |
| **date** | Date (ISO8601 в JSON) | да | Момент события (встал / сел). |
| **type** | String | да | Тип события: `"stand"` (встал) или `"sit"` (сел). |
| **durationMinutes** | Int? | нет | Только для `type == "stand"`: сколько минут простоял. Заполняется при переходе «сесть» для последней записи «встал». |

### Связи

- **type** — определяет смысл записи: `stand` или `sit`; у `sit` поле `durationMinutes` не используется.
- **date** — используется для отображения хронологии и для подсчёта статистики за день (в коде день берётся отдельно через `dayKey()`).
- Записи в массиве упорядочены от новых к старым (insert at 0); при очистке истории ключ `journalEntries` перезаписывается, связанные счётчики дневной статистики и серии обнуляются через `clearAllHistory()`.

## 2. Состояние таймера (Timer State)

**Ключи в UserDefaults:** `timer_phase`, `timer_phaseEndDate`, `timer_paused`  
**Код:** `StandUpState.swift` — `saveTimerState()`, `restoreTimerState()`.

Восстановление при запуске приложения: по этим ключам восстанавливаются фаза, время окончания и пауза, затем пересчитывается `remainingSeconds` и при необходимости показывается напоминание или запускается таймер.

### Поля (ключи)

| Ключ | Тип значения | Описание |
|------|----------------|----------|
| **timer_phase** | String | Текущая фаза: `"sitting"` или `"standing"` (enum `StandUpPhase.rawValue`). |
| **timer_phaseEndDate** | Double | Unix timestamp (TimeIntervalSince1970) момента окончания текущей фазы. По нему считается оставшееся время. |
| **timer_paused** | Bool | Таймер на паузе (true) или идёт (false). |

### Связи

- **timer_phase** — определяет, идёт ли отсчёт до напоминания «встать» (`sitting`) или до окончания стояния (`standing`).
- **timer_phaseEndDate** — одна точка во времени; связь с настройками: длительность фазы задаётся `reminderIntervalMinutes` (сидение) и `standDurationMinutes` (стояние) из настроек, при сохранении состояния записывается уже вычисленная дата окончания.
- Отдельных ключей для «связи с журналом» нет: журнал пишется при действиях пользователя (встал/сел), таймер только управляет фазами и напоминаниями.

## 3. Дневная статистика (Daily Stats)

**Ключи в UserDefaults:** `stands_<date>`, `standingMin_<date>`, `streakDays`, `lastStreakDay`, `lastResetDay`  
**Формат даты:** `yyyy-MM-dd` (например `2025-03-16`).  
**Код:** `StandUpState.swift` — `dayKey()`, `loadDailyStats()`, `saveDailyStats()`, `recordStandCount()`, `addActualStandingMinutes()`, `updateStreak()`, `resetIfNewDay()`, `clearAllHistory()`. `resetIfNewDay()` вызывается при запуске приложения и при открытии попапа (MainPopupView.onAppear), чтобы при смене календарного дня обнулять «стояний сегодня» и «стоя за день».

### Поля (ключи и значения)

| Ключ | Тип значения | Описание |
|------|----------------|----------|
| **stands_&lt;date&gt;** | Int | Количество «вставаний» за день с датой `date`. Увеличивается при каждом переходе в фазу стояния (встал). |
| **standingMin_&lt;date&gt;** | Int | Суммарные минуты стояния за день `date`. Накапливается при завершении фазы стояния (сел или таймер стояния истёк). |
| **streakDays** | Int | Текущая серия дней подряд с хотя бы одним вставанием. |
| **lastStreakDay** | String | Дата последнего дня, учтённого в серии, в формате `yyyy-MM-dd`. Нужна для расчёта streak при следующем вставании. |
| **lastResetDay** | String | Дата последнего дня, для которого считалась «дневная» статистика (`yyyy-MM-dd`). Используется в `resetIfNewDay()`: если текущий день по календарю отличается от `lastResetDay`, счётчики «сегодня» обнуляются и `lastResetDay` обновляется. Так избегают показа вчерашних цифр за сегодня (после перезапуска или если приложение было открыто через полночь). |

### Связи

- **stands_&lt;date&gt;** и **standingMin_&lt;date&gt;** привязаны к одной и той же дате `date`; для текущего дня ключ совпадает с `dayKey()` (текущая дата). При смене дня (`resetIfNewDay`) сравнивается текущий день с **lastResetDay**; при отличии счётчики обнуляются и записывается новый ключ для сегодня.
- **streakDays** и **lastStreakDay** связаны между собой:
  - при вставании: если `lastStreakDay` = вчера → `streakDays += 1`; если `lastStreakDay` не сегодня и не вчера → `streakDays = 1`; затем `lastStreakDay := сегодня`, `streakDays` сохраняется.
- С журналом: журнал не используется для пересчёта этих ключей; счётчики обновляются по действиям в реальном времени. При `clearAllHistory()` дневные ключи для текущего дня и серия обнуляются (в т.ч. удаляется `lastStreakDay`).

## 4. Настройки (Settings)

**Ключи в UserDefaults:** см. таблицу ниже.  
**Код:** `SettingsStore.swift` — enum `Keys`, инициализация и `didSet` у каждого свойства.

Все значения читаются при создании синглтона `SettingsStore` и сохраняются при каждом изменении свойства.

### Поля (ключи, типы, значения по умолчанию)

| Ключ | Тип | По умолчанию | Описание |
|------|-----|--------------|----------|
| **reminderIntervalMinutes** | Int | 60 | Интервал напоминания «встать», минуты. Варианты: 30, 60, 90. |
| **standDurationMinutes** | Int | 20 | Длительность стояния, минуты. Варианты: 10, 20, 30. |
| **soundEnabled** | Bool | true | Включён ли звук при напоминании и при окончании стояния. |
| **launchAtLogin** | Bool | true | Запуск приложения при входе в систему (через ServiceManagement). |
| **workingHours** | String | `"9:00-19:00"` | Режим рабочих часов: `"Всегда"`, `"8:00-18:00"`, `"9:00-19:00"` (enum `WorkingHoursPreset.rawValue`). В нерабочие часы напоминания не планируются. |
| **coloredIcon** | Bool | true | Цветная иконка в меню-баре (иначе серая). |
| **autoStartSittingTimer** | Bool | true | После окончания стояния автоматически запускать таймер сидения; иначе — остановить на 00:00 и мигать в трее. |

### Связи

- **reminderIntervalMinutes** и **standDurationMinutes** задают длительности фаз таймера; при восстановлении таймера из `timer_phase` и `timer_phaseEndDate` они не пересчитываются — используется уже сохранённая дата окончания.
- **workingHours** учитывается при планировании следующего уведомления (напоминания не ставятся вне выбранного окна).
- Остальные ключи настройки влияют только на поведение UI и системных уведомлений, прямых связей с другими «таблицами» не имеют.

## Диаграмма связей (логическая)

```
                    ┌──────────────────┐
                    │    Settings      │
                    │ (все ключи без   │
                    │  префикса даты)  │
                    └────────┬─────────┘
                             │ задают длительности фаз,
                             │ рабочие часы, звук и т.д.
                             ▼
┌─────────────┐     ┌──────────────────┐     ┌─────────────────────────┐
│   Journal   │     │  Timer State    │     │   Daily Stats           │
│ (массив     │     │ timer_phase     │     │ stands_<date>           │
│  Entry)     │     │ timer_phaseEnd  │     │ standingMin_<date>     │
│             │     │ timer_paused    │     │ streakDays             │
│ id, date,   │     └────────┬────────┘     │ lastStreakDay          │
│ type,       │              │              └───────────┬─────────────┘
│ durationMin │              │ восстановление при       │
└──────┬──────┘              │ запуске, обновление      │ привязаны к дате
       │                    │ при действиях            │ <date> = yyyy-MM-dd
       │                    │                          │ streakDays ↔ lastStreakDay
       │                    │                          │
       └────────────────────┴──────────────────────────┘
              журнал пишется при «встал»/«сел»;
              дневные счётчики обновляются там же
              (не из журнала)
```

Итого: одна «таблица» журнала (массив записей), три ключа состояния таймера, четыре группы ключей дневной статистики (с привязкой к дате и серии) и семь ключей настроек. Связи — через общую дату дня, через использование настроек при работе таймера и через общие действия пользователя (встал/сел), а не через внешние ключи в данных.
